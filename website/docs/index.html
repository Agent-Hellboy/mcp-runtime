Quick start

These commands are the current short path from a fresh repo checkout to a running platform flow.

make deps && make build-runtime

./bin/mcp-runtime setup
./bin/mcp-runtime status

# Optional:
# ./bin/mcp-runtime setup --with-tls
# ./bin/mcp-runtime setup --without-sentinel

./bin/mcp-runtime registry push --image my-server:latest
./bin/mcp-runtime pipeline generate --dir .mcp --output manifests/
./bin/mcp-runtime pipeline deploy --dir manifests/

Platform at a glance

The docs follow the same separation of concerns the codebase already uses.

Control plane

The runtime CLI and operator manage cluster setup, image flow, MCPServer deployment, ingress, and rollout behavior.

Access model

MCPAccessGrant and MCPAgentSession separate human and agent access, trust ceilings, consent, and revocation from deployment config.

Gateway path

Gateway mode moves traffic through the proxy sidecar so policy is enforced at tool call time and audit events are emitted into sentinel.

Analytics plane

Sentinel owns ingest, processor, API, UI, gateway, ClickHouse, Kafka, and observability services around the MCP servers.

Current scope

Deployment, routing, grants, sessions, gateway policy, and audit flow are implemented in the current repo. Sentinel supports bearer-token validation, but a full OAuth 2.1 authorization server and Dynamic Client Registration flow are still out of scope.

MCP Runtime docs track the current repo surface for the alpha platform. Use the runtime and API pages as the closest contract while the product evolves.
